🔷 vLLM은 이제 단순 서빙 엔진이 아니다
vLLM은 원래 PagedAttention 기반의 단일 노드 서빙 엔진으로 시작했습니다.
기존 inference 방식이 GPU 메모리의 60~80%를 낭비하던 문제를, OS의 가상 메모리 개념으로 해결한 것이 핵심이었죠.
하지만 지금 Production 현장의 vLLM은 그 이상입니다.
공식 Production Stack이 출시되면서 아래 4계층 구조가 하나의 Helm chart로 k8s에 배포 가능해졌고,
→ Serving Engine
→ Request Router
→ Observability (Prometheus + Grafana)
→ Autoscaling (KEDA)
LMCache를 통한 KV cache 공유와 prefix-aware routing으로 인스턴스 간 레이턴시 최적화까지 다루고 있습니다.
이번 밋업에서 반복적으로 등장한 또 다른 키워드는 CXL이었습니다. LMCache가 소프트웨어 레벨에서 KV cache 문제를 다룬다면, CXL은 메모리 아키텍처 단에서 GPU VRAM의 한계 자체를 허무는 방향입니다. 소프트웨어와 하드웨어가 같은 방향으로 수렴하고 있다는 느낌을 받았습니다.
단순히 모델을 띄우는 게 아니라, inference infrastructure를 어떻게 설계하느냐의 문제가 된 거죠.
---
🔷 보안 제약 안에서 만드는 사내 AI 환경 — 삼성전자 김성수님
가장 인상 깊었던 세션이었습니다.
반도체 특성상 외부 모델로 쿼리를 보낼 수 없는 제약 속에서, vLLM 기반 온프렘 서빙으로 사내 AI 환경을 구축한 사례였습니다.
OpenWebUI + Keycloak SSO 기반 실명제와 라운드로빈 로드밸런싱으로 사용량을 관리하고, Claude Code 같은 코딩 에이전트도 외부 API 없이 사내 오픈소스 모델로 대체해 운영하고 있었습니다. Dify와 n8n으로 에이전트를 만들어 사내에 직접 배포하는 사례들도 공유해주셨습니다.
보안 제약 안에서도 오픈소스 모델 기반의 서비스 배포와 워크플로우를 내부에서 구현할 수 있다는 것, 그 실제 사례를 직접 들을 수 있었던 자리였습니다.
