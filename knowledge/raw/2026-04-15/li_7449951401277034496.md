## 원문

“토큰을 많이 쓰는 사람이 더 생산성이 높다.”
“엔지니어는 연봉의 절반을 토큰에 써야 한다.”
요즘 자주 들리는 이야기입니다.
이 얘기를 듣고 나서, 하루에도 몇 번씩
제가 얼마나 토큰을 쓰고 있는지 확인하고 있는 제 모습을 발견했습니다.
만약 이 가설이 맞다면, 토큰 = 생산성이라면,
우리는 어떻게 더 많은 토큰을 “잘” 쓸 수 있을까? 지금 꽤 많이 생각하는 부분입니다.
돈을 많이 버는 사람들은 일을 해서 버는 것이 아니라
잘 때도 돈이 벌리는 구조를 만듭니다.
그렇다면, 디자이너도 잘 때도 토큰을 쓸수있는 구조를 만들수 있지 않을까?
그래서 제가 요즘 실험하고 있는 것이 있습니다.
“Sleeping Agent for Designers”
디자이너가 자는 동안, 생산성이 높은 토큰을 쓸수있을까?
제가 실제로 돌려보고 있는 5가지입니다.
1️⃣ 디자인 시스템 & 브랜드 가이드 자동 평가
아직 모든 회사가 사용할 수 있는 100% 자동화 솔루션은 없습니다. 그래서 저는 내부 디자인 시스템을 기반으로 다음 루프를 만들었습니다.
1. 새로운 UI 생성
2. 디자인 시스템 기준으로 자동 평가 (skills)
3. 실패 지점 분석
4. skills 업데이트
5. 기준 점수 미달 시 다시 생성
👉 이 과정을 95점 이상이 나올 때까지 반복
이 루프를 매일 밤 돌린 결과, 지금은 한 번에 95점 이상이 나오는 수준까지 올라왔습니다.
👉 아침에는 더 정교해진 skills와 리포트를 받습니다.
2️⃣ 퍼소나 기반 AI 프로덕트 응답 최적화
내가 만든 AI 제품의 응답은 정말 최적인가?
사전에 정의한 퍼소나를 기반으로 응답을 계속 실행 → 평가 → 개선합니다.
👉 만족도 99%에 도달할 때까지 반복
👉 아침에는 더 정교한 프롬프트와 리포트를 받습니다.
3️⃣ 코드 리팩터링 + QA
디자이너에게 익숙하지 않은 영역일 수 있습니다. 하지만 프로토타입이 커질수록 필수입니다.
- 코드 정리
- 버그 탐지
- 안전한 수정
👉 밤 동안 반복 수행
👉 아침에는 더 안정적이고 정돈된 코드와 리포트를 받습니다.
4️⃣ 퍼소나 기반 유저 시뮬레이션
다양한 사용자 입장에서 제품을 실제로 사용하게 합니다.
어디서 막히는지, 어디서 헷갈리는지, 어떤 edge case가 있는지
👉 약 100회 이상 반복 실행 후 통계 확보
중요한 점:
디자인 변경은 하지 않습니다. (HITL: Human in the loop 영역)
👉 아침에는 시뮬레이션 리포트와 개선 제안을 받습니다.
5️⃣ Service Metrics 기반 인사이트 생성
- 유저 피드백 (qualitative)
- Web / App Metrics (quantitative)
- KPI
- QA 결과
이 모든 데이터를 연결해서 “왜”를 설명합니다.
단순 데이터가 아니라 이렇게 나옵니다:
“Step 3에서 이탈이 높은 이유는
CTA가 명확하지 않고, 불필요한 단계가 있기 때문입니다.”
👉 데이터 → 인사이트 → 방향성
제가 속한 상황에서는 필요한 에이전트 들인데 다른 디자이너분들에게는 그닥 필요없는 에이전트일수도 있습니다. 그러면 생각해보세요,
여러분이라면 잘 때 어떤 에이전트를 돌릴수있을까요?
