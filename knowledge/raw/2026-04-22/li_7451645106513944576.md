두 회사 모두 하네스도 쓰고 멀티에이전트도 쓰지만 접근 방식은 좀 달랐습니다.

✅ 하네스 — 맥락을 어디에, 어떻게 쌓나
Meta는 "팀 단위 쿡북"으로, OpenAI는 "리포지토리"로
[Meta: 쿡북 / 레시피 / 재료 3단]
사용자 7만 명 데이터 웨어하우스 규모를 감안해,
Cookbook → Recipe → Ingredient 구조를 사용합니다
→ Cookbook: 팀·도메인 단위 번들. 레시피 + 재료 + 비즈니스 맥락 + 프롬프트
→ Recipe: 어떻게 분석할지 표준 분석 절차,
시니어 분석가의 SQL 히스토리를 Reference와 Expert로 걸고, 검증 규칙으로 사용
→ Ingredient: '데이터가 뭘 뜻하는지'의 시맨틱. 예) l7_active = 지난 7일 활동,
이탈 계정 제외, 국가, 날짜 단위 집계(grain)
의외였던 건 이 설계의 출발이 된 관찰이었습니다. 데이터 사이언티스트 쿼리의 88%가
최근 90일 안에 봤던 테이블을 재사용한다고 합니다. 7만 명 분석 조직에서도 분석
대부분은 반복이라는 거죠... 그 반복 구조를 쿡북에 고정시키니 비데이터 직군까지
DS,DE 대비 5배 규모로 쓰게 됐다고 합니다. 베타에만 커뮤니티 레시피 4,500개가
만들어져 15만 번 쓰였다는군요. '큰 조직은 우리랑 다른 걸 하겠지'라고 막연히
생각했는데, 반복 패턴 자체는 규모가 달라도 비슷하다는게 와닿았어요.
[OpenAI: 리포지토리가 system of record]
OpenAI는 에이전트가 접근할 수 없는 지식은 없는 것과 같다고 봅니다.
슬랙,구글독스, 머릿속 지식은 사람에겐 있지만 에이전트에겐 없다는 거죠.
그래서 모든 맥락을 리포지토리에 박아둡니다.
→ docs/ = system of record. 백과사전이 아니라 목차로 취급
→ AGENTS.md = 지도. 짧게 유지하고 더 깊은 출처를 가리키는 포인터
3,500명 이상의 인원이 내부 데이터 에이전트를 사용하며, 슬랙·웹·IDE·Codex CLI·내부 ChatGPT까지 제공합니다. 어디서 일하든 에이전트가 옆에 붙어 있는 구조예요.
——
✅ 멀티에이전트 — 역할을 어떻게 나누나
Meta는 분석 파이프라인 안에서, OpenAI는 엔지니어링 전체에서 풉니다.
[Meta]
- Tribal knowledge 추출: 데이터 파이프라인에 묻혀 있던 암묵지를 꺼내기 위해 50개 이상의 전문 에이전트를 배치했습니다. 각 에이전트가 모듈마다 정해진 5가지 질문(이 모듈은 뭘 설정하나, 어떤 변경 패턴이 있나 등)에 답변합니다.
- 검증 분리: 레시피의 Custom Validation은 별도 AI가 검증합니다.
[OpenAI]
작업하는 에이전트와 정리하는 에이전트를 분리했습니다.
