## 원문

[에이전트를 이해하려면 인간에게 대응시켜보세요.]
최근 어딜가도 에이전트를 활용해서 자동화를 했다는 말이 들리고,
심지어는 어제 클로드코드 소스코드까지 유출되면서 에이전트에 대한 FOMO성 궁금증이 많아진 것 같다.
뭔가 직접 알아보려고 하면 직면하는 단어들이
model, CLAUDE.md, tool-call, mcp, cli, skill, sub-agent, agent-team, orchestration, harness 이런식이라서 다들 어려움을 느끼는 것 같다.
이런 FOMO와 어려움을 이용한 강의팔이가 성행하는데 그러고 싶지는 않다.
하지만 한국에서 에이전트 이해도로 열손가락...은 모르겠고 백손가락 안에는 든다고 자신하며, 어제 유출된 클로드코드 소스코드를 뜯어보면서 이 방식대로 클로드코드가 만들어지고 있다는 확신을 가졌기에, 내가 에이전트를 이해하는 방법을 공유해보고자 한다.
제목에 말했듯이 모든걸 인간에 대응시키면 이해하기가 훨씬 쉽다.
model은 지능 수준이다. 안좋은 모델은 머리가 좋지 않은 사람을 고용한 것이고, 좋은 모델은 고지능의 사람을 고용한거라고 생각하면된다.
CLAUDE.md는 업무 메뉴얼이다. 요즘 CLAUDE.md를 많이 쓰면 별로다 이런 말이 있는데, 이 조차 당연하다. 이제 막 입사한 신입사원에게 몇십장짜리 업무메뉴얼을 주면 그게 소용이 있겠는가. 핵심만 모아서 딱 '아 나는 이런걸 해야하는 거구나' 정도를 알 수 있도록 적혀있는게 best이고, Agent에게도 동일하다.
tool-call은 실제 작업 단위이다. 예를들어 어떤 개발자가 기획서를 보고 기능을 실구현하는 일을 쪼개본다면 (기획서 열람 -> 기획 이해 -> 구현 계획 -> 각 파일별로 실제 코딩) 이렇게 될 것 이다. 이 작업단위 하나하나가 Agent에서는 하나의 tool-call로 일어나는 것이다.
mcp/cli는 Agent가 할 수 있는 실제 작업 단위를 알려주는 것이다. 방금 예시에서 그 기획서가 Notion에 쓰여져있었다면 사람은 노션을 켜서 기획서를 볼 것이다. Agent는 특별한 주입이 없다면 Notion에서 기획서를 보면된다는 것을 알 수가 없기에, "Notion에서 기획서 보기" 라는 작업 단위가 있음을 알려주는 것이다.
skill은 작업 가이드라인이다. 예시 수준 업무라면 사람도 가이드라인 없이 알아서 잘 쪼갤 것이고, Agent도 동일하다. 하지만 예시보다 훨씬 복잡한 작업이나, 특정 단계에서 조심해야하는 포인트가 있다면 사람에게도 가이드라인이 필요할 것이다. Agent도 동일하고 꼭 복잡해서 뿐 아니라 자주 반복되는 작업이라면 가이드라인화 해놔서 생각많이 안하고 처리할 수도 있는 것이다.
어려운 일, 반복되는 일은 가이드라인이 있으면 더 정확하고 신속하게 처리되므로 가이드라인을 만들면 좋다! 라는 것이 사람이 일하는 방식과 똑같다.
추가적으로 mcp 구리고 cli + skill이 낫다! 이런말이 있는데 이것은 이런 느낌이다. 기획을 알기위해선 Notion만 보면되는데 mcp는 내가 할 수 있는 작업을 항시 주입해준다. 즉, 여러 mcp를 붙여놓은 경우 "notion에서 기획서보기"를 할 수 있다는 것만 알면되는데 "figma에서 디자인보기", "slack에서 대화보기"를 할 수 있다고 계속 알려주고 있는 것이다. 반면 cli + skill은 가이드라인에 "기획서는 노션에 있고 노션은 이 cli 명령어로 찾아서 읽으면 된다." 이렇게 적혀있으면 skill이 trigger됐을때 그 cli로 기획서를 읽는다.
즉, 능동성 차이라고 보면된다. 개인적인 의견으로는 cli + skill이 더 예측 가능해서 좋다.
sub-agent, agent-team은 협업이다. sub-agent는 팀장이 팀원에게 작업하달 후 결과만 정리받는 단방향성 협업이고, agent-team은 동일한 직급의 팀원이 서로 상의하는 양방향성 협업이다.
Agent의 협업구조 또한 인간의 형상과 동일한데, 쓸데없이 sub-agent를 많이 만들거나 어렵지 않은 작업을 agent-team으로 하면 시간만 길어지고 비효율적이다. 이건 마치 규모가 작은 스타트업은 올라운더가 많이 있고, 규모가 큰 기업은 세세하게 직무가 나뉘어져 있는 것과 같다. 업무의 양과 크기에 따라 적절히 agent 협업 규모를 세팅하는게 중요하다.
orchestration/harness는 회사시스템이다. 신상품 런칭 과정을 회사 레벨에서 바라보면, 기획/리서치팀은 신상품 기획을 할 것이고 재무팀 등 검토팀은 그걸 검토할 거고 최종승인이나면 마케팅/컨텐츠/생산팀으로 일이 전파 될 것이다. 이렇게 경영의 레벨에서 agent가 굴러가도록 만드는 것이라고 생각하면 된다.
이렇게 Agent를 이해하기위해 필요한 주요 개념들을 사람에 빗대어 정리해보았다.
눈치챘을지 모르겠지만 재밌는 점이 있다. 글의 작성 순서가 개념 등장 순서와 동일한데
처음에는 Model. 일단 똑똑한 개인을 만드는데 집중했다.
다음에는 CLAUDE.md, tool-call, mcp/cli/skill. 개인이 똑똑해지고 나니 개인이 일을 정확하게 수행하는 방법을 만들었다.
다음은 sub-agent, agent-team. 개인이 일을 정확하게 하니 협업을 고민했다.
이제 그 다음은 orchestration/harness. 협업까지 되니 경영을 고민한다.
물론 memory등 설명하지 않은 개념도 많지만, 이것도 동일하게 사람에게 대응시키면 당연히 필요한 개념이다. 근데 유출된 클로드 코드 소스에서 Anthropic도 나와 동일하게 생각하고 있다고 확신을 가졌다고 했는데, 그 이유가 memory에 망각기능을 붙이고 있음을 확인해서다. 결국 사람은 단기기억에서 잊을 건 잊고 장기기억으로 넘긴다.
이렇게 Agent의 다음 발전방향을 고민할때는 결국 사람은 어떻게 사고하는가? 를 추상화 해보면 답이 나오는 것 같다.
처음엔 이렇게까지 알찬 글을 쓰려던 건 아니었지만 퇴고해보니 내용이 굉장히 알찬 것 같다. Agent를 이해하는데 도움이 됐다면 따봉 하나씩이라도 주면 고마울 것 같다.
