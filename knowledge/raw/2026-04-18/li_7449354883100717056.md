## 원문

[ AI 디자인시스템 인프라 구축기 #2 ]
한 달 전 쯤, AI 프로토타이핑 툴에 내부 디자인 시스템을 범용적으로 적용하는 인프라를 구축하고 있다고 글을 올렸습니다. 그 과정과 현시점 결과에 대한 회고 내용입니다.
답답하기도 했고, 가끔은 고독하기도 했습니다😢
'이건 아직 안 된다'는 말을 혼자 반복하면서도, 직접 부딪혀보지 않으면 설득할 수 도 없는 것들이니까요,
FIgma를 버리고, AI 프로토타이핑 툴로 우리 디자인 시스템을 그대로 구현할 수 있을까?
목표 : 정합도 80%이상
결과 : 정합도 65~70%
시도한 방법은 3가지입니다.
1. MD 파일 방식
v0, Lovable 등 어떤 툴이든 읽을 수 있도록 디자인 시스템 규칙을 텍스트로 설명하는 방식입니다. 레이아웃과 색상은 어느 정도 따라오지만, 실제 컴포넌트는 AI가 새로 만들어버립니다. 우리 Button을 쓰는 게 아니라 Button처럼 생긴 걸 만들어버립니다.
2. GitHub JSON Registry
v0에서 디자인 시스템을 불러올 수 있도록 Registry를 직접 만들었습니다. CSS 토큰과 폰트 정도는 적용되지만, v0가 반복적으로 자체 색상 시스템으로 덮어쓰는 문제가 있었습니다.
3. Vercel Registry Starter (shadcn 기반)
현재까지 내부 shadcn커스텀 디자인시스템에 가장 잘 작동하는 방식입니다. CSS 토큰과 컴포넌트 등록까지 성공했고, v0가 디자인 시스템을 인식합니다. 그럼에도 import 경로는 여전히 자체 생성하는 경우가 있었습니다.
이 과정에서 명확해진 것이 있습니다.
MD 파일은 영양 보조제입니다. Registry나 npm이 실물 컴포넌트를 붙여주는 '식사'라면, MD 파일은 '어떻게 먹어라'를 알려주는 보조제예요. 아무리 잘 만들어도 실물 없이는 한계가 있습니다.
Figma를 사용하지 않고, AI 프로토타입에서 개발 핸드오프는 현재 시점에서 어렵다고 생각합니다. AI가 만든 결과물의 프리뷰는 그럴듯해 보입니다. 하지만 코드를 열어보면 import 경로가 전부 다르고, 우리 컴포넌트는 하나도 쓰이지 않은 경우가 대부분입니다. 정합도 80% 이상은 npm 배포 없이는 구조적으로 어렵습니다.
그리고 비로소 최근 조직 내부에서도 같은 결론에 도달했습니다.
혼자 실험하고 테스트하면서 정합도 90%이상에 도달하지 못해 답답하지만 과정에 대한 가치가 있다고 생각합니다.
AI 툴은 빠르게 발전하고 있어서 6개월~1년 후엔 달라질 수 있습니다. 하지만 지금 이 시점에서, 'AI로 디자인 시스템 100% 정합도'는 아직이라고 생각됩니다.
