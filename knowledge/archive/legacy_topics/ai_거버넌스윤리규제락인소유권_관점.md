# AI 거버넌스/윤리·규제(락인/소유권 관점)

> 업데이트: 2026-04-15  
> 출처: LinkedIn 포스트 분석 통합  
> - 2026-04-15, LinkedIn post by 관련 작성자(포스트 내 LangChain CEO Harrison Chase 논의 인용), https://www.linkedin.com/feed/update/urn:li:activity:7449569619729629184/

## 주제 개요

생성형 AI와 AI Agent 도입이 확산되면서, 기업의 거버넌스 이슈는 단순한 모델 선택을 넘어 **누가 컨텍스트(context), 메모리(memory), 스킬(skill)을 소유·이동·통제하는가**로 이동하고 있다.  
특히 Agent 아키텍처에서는 경쟁우위의 핵심이 모델 자체보다 **업무 맥락과 축적된 실행 지식**에 있다는 관점이 부상하고 있으며, 이는 곧 벤더 락인(lock-in), 데이터 소유권, 이식성(portability), 감사 가능성(auditability) 문제와 직결된다.

---

## 핵심 쟁점

### 1. AI Agent 시대의 락인 대상이 모델에서 컨텍스트/메모리로 이동
**사실**
- 포스트는 Agent 시스템의 학습/가치 축적 레이어를 `Model`, `Harness`, `Context`의 3개로 구분한다.
- 이 중 실질적 학습과 차별화가 `Context` 레이어에서 일어난다고 설명한다.
- `Context`에는 명령어, 스킬, 메모리, 업무별 축적 지식이 포함된다.

**해석/의견**
- 기업 관점에서 장기 경쟁우위는 모델 성능 그 자체보다, 조직이 축적한 **업무 문맥과 실행 패턴**에 있을 가능성이 높다.
- 따라서 향후 락인 리스크는 “어느 LLM을 쓰는가”보다 “축적된 컨텍스트를 다른 실행 환경으로 옮길 수 있는가”가 더 중요해질 수 있다.

### 2. 오픈 vs 폐쇄보다 중요한 것은 ‘실질적 소유권’
**사실**
- Harrison Chase의 논지는 “진정한 memory ownership은 open harness architecture를 필요로 한다”는 방향으로 소개된다.
- 반면 포스트 작성자는 폐쇄형 harness를 사용하더라도 메모리와 스킬이 **로컬 파일 기반**으로 관리되면 이식성과 소유권을 유지할 수 있다고 주장한다.
- 예시로 `CLAUDE.md`, `AGENTS.md`, `SOUL.md` 등 서로 다른 Agent 실행 환경이 유사한 패턴으로 컨텍스트 파일을 읽는 구조가 제시된다.
- 작성자는 자신의 스킬과 메모리를 로컬 마크다운 파일로 저장하고 `git`으로 버전 관리한다고 설명한다.

**해석/의견**
- 락인 문제를 “오픈 소스냐, 폐쇄형 제품이냐”의 이분법으로만 보면 불충분하다.
- 더 본질적인 질문은 다음과 같다.
  - 컨텍스트와 메모리가 **벤더 DB**에만 저장되는가?
  - 기업이 이를 **파일/저장소 단위로 직접 보관**할 수 있는가?
  - 다른 harness나 Agent 프레임워크로 **이전 가능**한가?

### 3. 메모리 소유권은 거버넌스 이슈이자 운영 설계 이슈
**사실**
- 포스트는 폐쇄형 harness라도 로컬 파일 기반 메모리 구조를 채택하면 벤더 종속을 줄일 수 있다고 본다.
- 반대로 오픈 harness라도 메모리가 벤더 또는 특정 플랫폼 DB에 종속되면 실질적 소유권이 약해질 수 있다는 문제의식을 제기한다.

**통합 인사이트**
- 메모리 소유권은 라이선스 정책만의 문제가 아니라, 실제 저장 구조와 운영 방식의 문제다.
- 즉, AI 거버넌스는 정책 문서뿐 아니라 아래와 같은 **기술 아키텍처 선택**까지 포함해야 한다.
  - 메모리 저장 위치
  - 포맷 표준화 여부
  - 버전 관리 체계
  - 백업/복구 가능성
  - 접근권한 및 변경 이력 추적

## 주요 관점 정리

### Harrison Chase 관점
**사실**
- Agent 시스템의 핵심 가치가 `Context/Memory` 레이어에 있다고 본다.
- 메모리 소유권을 위해서는 `open harness architecture`가 필요하다고 주장하는 것으로 소개된다.

**의미**
- Agent 플랫폼 경쟁이 모델 API 경쟁에서 **메모리/컨텍스트 실행 환경 경쟁**으로 이동하고 있음을 보여준다.
- LangChain/Deep Agents 계열 포지셔닝과도 연결된다.

### 포스트 작성자 관점
**사실**
- 실제 업무에서 60개 이상의 스킬을 축적해 사용 중이라고 설명한다.
- 메모리와 스킬을 로컬 마크다운 파일로 저장하고, 여러 harness 간 재사용 가능하다고 주장한다.

**의견**
- 진짜 핵심은 오픈/폐쇄 여부보다 **내 context를 내가 갖고 있는가**이다.
- 로컬 파일 + 버전 관리 + 포터블한 구조가 실질적 소유권을 보장하는 더 현실적인 기준이라는 입장이다.

## 기업 AX 관점에서의 시사점

### 1. Agent 도입 시 ‘모델 선정’보다 ‘컨텍스트 자산화 전략’이 중요
기업은 LLM 성능 비교에만 집중하기 쉽지만, 실제 운영 경쟁력은 아래 자산의 축적 여부에서 갈릴 가능성이 높다.
- 업무별 프롬프트/지침
- 역할별 스킬 라이브러리
- 고객/파트너/프로젝트별 메모리
- 반복 업무의 실행 패턴
- 실패/피드백 이력

이 자산이 벤더 내부에만 축적되면, 향후 전환 비용이 급격히 커질 수 있다.

### 2. 메모리와 스킬은 기업의 지식재산(IP)로 다뤄야 함
Agent의 메모리와 스킬은 단순 설정값이 아니라, 조직이 시행착오를 통해 축적한 운영 지식이다.  
따라서 다음과 같이 관리할 필요가 있다.
- 문서 자산으로 분류
- 저장소 정책 수립
- 버전 관리 적용
- 퇴사/이관/감사 시 인수인계 대상 포함
- 민감정보 포함 여부에 따른 등급 분류

### 3. 거버넌스 프레임을 ‘데이터 소유권’에서 ‘컨텍스트 소유권’으로 확장해야 함
기존 AI 거버넌스는 주로 학습 데이터, 개인정보, 출력물 저작권에 초점을 맞췄다.  
Agent 환경에서는 여기에 더해 다음이 필요하다.
- 세션 간 메모리의 소유권
- Agent가 축적한 업무 문맥의 이전 가능성
- 스킬/워크플로우의 재사용 권리
- 벤더 종료 시 반출 권리와 포맷 보장

## 거버넌스 체크리스트

### 아키텍처
- 메모리가 벤더 SaaS 내부 DB에만 저장되는가?
- 로컬/사내 저장소로 내보내기(export)가 가능한가?
- 컨텍스트 파일이 표준적이고 사람이 읽을 수 있는 포맷인가?
- 다른 harness/Agent 프레임워크로 이전 가능한가?

### 통제
- 누가 메모리를 생성·수정·삭제할 수 있는가?
- 변경 이력과 버전 추적이 가능한가?
- 메모리/스킬별 접근권한이 분리되는가?
- 퇴사자/외주 인력의 접근권한 회수가 가능한가?

### 규제/리스크
- 메모리에 개인정보·고객기밀·계약정보가 포함되는가?
- 장기 저장 시 보존 기간과 삭제 정책이 있는가?
- 감사 시 어떤 컨텍스트가 의사결정에 사용됐는지 재현 가능한가?
- 벤더 계약 종료 시 데이터 반출 조항이 있는가?

### 운영
- 스킬과 메모리를 조직 공용 자산으로 관리할 것인가, 개인 자산으로 둘 것인가?
- 개인별 Agent 메모리가 조직 지식으로 전환되는 프로세스가 있는가?
- 업무별 템플릿과 naming convention이 정의되어 있는가?

## 트렌드 및 패턴

### 1. Agent 경쟁의 중심이 ‘실행 엔진’에서 ‘기억 자산’으로 이동
- 모델 성능 상향 평준화가 진행될수록, 차별화는 조직 특화 컨텍스트와 메모리 축적에서 발생할 가능성이 높다.

### 2. 파일 기반·포터블 메모리 구조에 대한 관심 확대
- 마크다운 기반 컨텍스트 파일, 로컬 저장, `git` 버전 관리 같은 방식이 실무적 대안으로 부상한다.
- 이는 개발자 친화적일 뿐 아니라 감사 가능성과 이식성 측면에서도 장점이 있다.

### 3. 오픈 아키텍처 담론의 재해석
- “오픈이냐 폐쇄냐”보다 “반출 가능하냐, 사람이 읽을 수 있냐, 다른 환경에서 재사용 가능하냐”가 더 실무적인 평가 기준으로 떠오른다.

## 실행 가능한 권고안

### 단기
1. 현재 사용 중인 AI 도구의 메모리 저장 위치를 파악한다.
2. Agent별 컨텍스트/스킬/메모리 자산 목록을 만든다.
3. 벤더별 export 가능 여부와 포맷을 점검한다.
4. 민감정보가 메모리에 저장되는지 확인한다.

### 중기
1. 컨텍스트 자산의 표준 포맷을 정의한다.
2. 로컬 또는 사내 저장소 기반의 메모리 관리 체계를 설계한다.
3. `git` 등 버전 관리 체계를 도입한다.
4. Agent 스킬과 메모리를 조직 공용 자산으로 전환하는 프로세스를 만든다.

### 장기
1. 멀티-harness/멀티-vendor 전략을 고려한 Agent 아키텍처를 설계한다.
2. 계약서에 메모리/컨텍스트 반출권, 삭제권, 감사권을 명시한다.
3. AI 거버넌스 정책에 ‘컨텍스트 소유권’ 조항을 신설한다.

## 한줄 결론

AI Agent 시대의 락인 핵심은 모델이 아니라 **조직의 context/memory를 누가 저장하고, 통제하고, 이전할 수 있는가**에 있다.  
따라서 기업의 AI 거버넌스는 **데이터 소유권**을 넘어 **컨텍스트 소유권과 메모리 이식성**까지 포함하도록 확장되어야 한다.
